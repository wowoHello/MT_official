# 資料庫正規化指南

資料庫正規化是設計關聯式資料庫的過程，旨在減少資料冗餘、提高資料完整性並優化資料庫結構。以下是主要的正規化範式：

## 第一正規化範式 (First Normal Form, 1NF)

- **定義**：資料表中的所有欄位都必須是原子性的，不可再分割。每個欄位都只包含單一值，且沒有重複的群組。
- **範例**：
  - **不符合 1NF**：一個訂單欄位包含多個商品資訊 (商品A, 商品B)。
  - **符合 1NF**：將訂單商品拆分為獨立的訂單明細表，每行只包含一個商品資訊。

## 第二正規化範式 (Second Normal Form, 2NF)

- **定義**：資料表必須先符合 1NF，且所有非主鍵欄位都必須完全依賴於整個主鍵。如果主鍵是複合鍵，則非主鍵欄位不能只依賴於主鍵的一部分。
- **範例**：
  - **不符合 2NF**：訂單明細表 (主鍵: 訂單ID, 商品ID) 中包含商品名稱和商品價格，而商品名稱和商品價格只依賴於商品ID。
  - **符合 2NF**：將商品資訊拆分為獨立的商品表，訂單明細表只保留商品ID。

## 第三正規化範式 (Third Normal Form, 3NF)

- **定義**：資料表必須先符合 2NF，且所有非主鍵欄位都不能存在遞移依賴 (Transitive Dependency)。即非主鍵欄位不能依賴於另一個非主鍵欄位。
- **範例**：
  - **不符合 3NF**：員工表 (主鍵: 員工ID) 中包含部門名稱和部門經理，而部門經理依賴於部門名稱。
  - **符合 3NF**：將部門資訊拆分為獨立的部門表，員工表只保留部門ID。

## BCNF (Boyce-Codd Normal Form)

- **定義**：比 3NF 更嚴格的正規化形式。資料表必須先符合 3NF，且所有決定因素 (Determinant) 都必須是候選鍵 (Candidate Key)。
- **適用時機**：當資料表有多個重疊的候選鍵時，BCNF 可以進一步消除冗餘。

## 反正規化 (Denormalization)

- **定義**：在某些情況下，為了提高查詢效能，可以有意識地引入冗餘，將部分正規化的資料表進行合併或增加重複欄位。
- **適用時機**：當查詢頻繁且複雜，且正規化導致過多的 JOIN 操作影響效能時。但需謹慎使用，並確保資料一致性。
