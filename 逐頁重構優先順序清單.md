## 判讀原則

本清單以三個面向排序：

1. 功能是否明顯未完成或與需求落差大
2. 是否存在未接上的舊程式碼、占位功能或暫代實作
3. 是否偏離既定 code 規則，導致頁面過胖、責任混雜、後續難維護

---

## P0：先處理

這一層不是「有點醜」，是已經會直接影響功能完整度或後續維護判斷。

### 1. Reviews

- 優先原因：
  - 頁面 UI 與實際資料接線明顯脫節。
  - `ReviewService.GetHistoryAsync(int projectId)` 有實作但完全沒有呼叫者，屬於明確未接上的邏輯。
  - Hope 需求中的搜尋、篩選、分頁、歷史區統計，目前大多還是靜態殼。
- 主要問題：
  - Tab badge 寫死 `0`，沒有綁資料。
  - 歷史區統計卡片也寫死 `0`。
  - 搜尋、題型、等級、狀態、每頁筆數 UI 存在，但沒有實際狀態與查詢邏輯。
  - 頁面只載入 `assignments`，沒有 history state。
- 建議動作：
  - 先補齊「審題作業區 / 歷史區」兩套資料流。
  - 決定 `GetHistoryAsync()` 是正式接上，還是刪除改寫。
  - 把篩選、統計、分頁抽成可共用的 view state。
- 相關位置：
  - `Components/Pages/Reviews.razor`
  - `Services/ReviewService.cs`

### 2. Announcements

- 優先原因：
  - 規則落差明確，尤其是置頂限制與使用說明手冊仍為占位。
  - 前端與後端對置頂規則沒有守死，容易留下資料不一致。
- 主要問題：
  - `ShowUserGuide()` 仍是 SweetAlert 假資料，不是實際手冊功能。
  - Hope 定義「僅已發佈可置頂」，目前前端允許 `已發佈 / 未發佈` 顯示置頂按鈕。
  - `TogglePinAsync()` 只做 DB toggle，沒有再次驗證狀態是否合法。
  - 切換置頂採前端樂觀更新後整批重載，對單一布林操作偏重。
- 建議動作：
  - 先決定手冊功能最終資料來源與檔案儲存方式。
  - 把置頂規則收斂到 service 層，前端只負責顯示。
  - 重新整理 pin 流程，避免先改 UI 再重載整頁。
- 相關位置：
  - `Components/Pages/Announcements.razor`
  - `Services/AnnouncementService.cs`

### 3. Projects

- 優先原因：
  - 頁面太胖，已開始靠 `@key` 強制重建 checkbox/select 來補狀態，這通常表示狀態模型打結了。
  - 後續再加規則，成本只會更高。
- 主要問題：
  - 人員勾選、角色 slot、配額分配都擠在同一頁 state。
  - `CheckboxKey` / `RoleSlotKey` 這種補救型欄位已經出現。
  - `.razor` 內嵌多個頁面專屬 model，沒有往 component/view model 拆。
- 建議動作：
  - 優先拆出「人員指派區」與「配額分配區」元件。
  - 把 `PersonItem` / `AllocationItem` / `StageItem` 等頁內模型移出。
  - 重整角色變更與命題教師保護規則，減少 UI 強制重建技巧。
- 相關位置：
  - `Components/Pages/Projects.razor`
  - `Services/ProjectService.cs`
  - `Models/ProjectModels.cs`

## P1：第二批整理

這一層多半功能能跑，但結構已經開始變重，越晚拆越難收。

### 4. Roles

- 優先原因：
  - 單頁同時處理帳號管理、角色管理、權限切換、角色使用者清單，責任過多。
  - 現在能跑，但不是好養的狀態。
- 主要問題：
  - `CreateAccountRequest` / `CreateRoleRequest` 直接拿來當 UI form state。
  - 帳號 modal 與角色 modal 都很重，且全塞在同頁。
  - 狀態欄位數量多，頁面心智負擔高。
- 建議動作：
  - 帳號區與角色區拆成獨立子元件。
  - 建立專用 form model，避免直接污染 request DTO。
  - 重新整理 modal 開關與 reset 流程。
- 相關位置：
  - `Components/Pages/Roles.razor`
  - `Services/RoleService.cs`
  - `Models/RoleModels.cs`

### 5. Teachers

- 優先原因：
  - 功能完整度不差，但頁面已承擔列表、詳情、歷程、表單、指派、匯出占位功能。
  - 再加需求很容易一起糊掉。
- 主要問題：
  - `HandleExport()` 仍是「功能開發中」占位提示。
  - 選教師後會同步載入詳情、命題歷程、審題歷程、參與專案，頁面責任偏重。
  - 歷程 tab、表單、加入梯次 modal 都在單一 `.razor`。
- 建議動作：
  - 先決定匯出功能是否本期要做；若不做，至少明確標記為 backlog。
  - 拆 `TeacherDetailTabs`、`TeacherFormModal`、`AssignProjectModal`。
  - 把詳情載入流程集中到單一 view model/service orchestration。
- 相關位置：
  - `Components/Pages/Teachers.razor`
  - `Services/TeacherService.cs`
  - `Models/TeacherModels.cs`

### 6. Dashboard

- 優先原因：
  - 主功能已成形，但稽核歷程與需求有明顯落差。
  - 頁面 helper 與圖表渲染責任偏重。
- 主要問題：
  - Hope 要求顯示登入 / 登出 / CRUD，現在查詢只抓 `Create / Update / Delete`。
  - 圖表組裝、稽核顯示、緊急項目 helper 都留在頁面。
- 建議動作：
  - 先補齊是否要納入登入/登出 log。
  - 逐步把圖表 mapping 與 log label helper 往 model/service 搬。
- 相關位置：
  - `Components/Pages/Dashboard.razor`
  - `Services/DashboardService.cs`
  - `Models/DashboardModels.cs`

## 重構原則建議

- 先補真正功能缺口，再做純結構美化。
- 大頁面優先拆「高互動區塊」與「Modal 表單區塊」。
- 不要直接拿 request DTO 當 UI form state。
- 規則型限制要回收到 service 層，不要只靠前端顯示控制。
- 只要頁面開始出現 `Key++` 這種強制重建 UI 的補救法，就該回頭整理 state model 了，這種通常不是魔法，是債。
